# 技术博客写作

好技术博客不是功能说明书，而是**用一个问题牵引读者、用经验支撑论点、用结构降低认知负担**的文章。核心原则：先回答「读者为什么要关心」，再展开「是什么 / 怎么做 / 何时不用」。

**不要用于**：纯新闻稿、版本发布公告、无技术深度的营销文。

## 文章类型

| 类型 | 读者诉求 | 核心交付 |
|------|----------|----------|
| 产品技术拆解 | 「它怎么做到的？」 | 架构、关键决策、权衡 |
| 领域技术深潜 | 「我该怎么用/理解？」 | 心智模型、模式、陷阱 |
| 经验复盘 | 「踩坑后学到了什么？」 | 根因、方案、可复用原则 |

选题自检：能否用一句话说清**读者读完后能做什么**？不能则收窄范围。

## 文章结构

```
1. 开篇钩子     — 工程问题 / 痛点 / 反直觉结论（1–2 段）
2. 定义与边界   — 术语、与相近概念对比、适用/不适用场景
3. 核心内容     — 由浅入深：基础块 → 模式 → 进阶
4. 实战/案例    — 代码、架构图、真实取舍
5. 常见陷阱     — 失败模式 + 修复
6. 收束         — 3 条以内原则，不重复正文
```

**产品拆解**：问题先行 → 分层展开 → 显式权衡（延迟 vs 准确率、简单 vs 灵活）→ 每个模式配 When/When not。  
**领域科普**：对比框（X vs Y）→ 一个类比贯穿全文 → 最小可行示例 → 原则收束。

## 写作技巧

**开篇**：首段承诺读者收益（能做决策或能避坑），不罗列目录。  
- 坏：「本文介绍 X 的功能。」  
- 好：「当 agent 能力增强，爆炸半径也在增大。工程问题是：如何限制它？」

**段落**：一段一个观点，段首句即论点；复杂流程用编号，并列选项用表格；代码块前写为何需要，后写输出与边界。

**论证**：生产经验作证据；描述两个极端及中间地带；数字要有语境（不说「提升 50%」，说「从 230 步降到 120 步」）。

**可视化**：优先顺序 ASCII 架构图 > 表格 > Mermaid，图必须服务论证。

**语气**：自信但谦逊，承认权衡；技术术语保留英文时首次中英并注；避免「强大的」「颠覆性的」等空洞形容词。

## 本项目发布格式

文章存于 `src/content/blog/*.mdx`，frontmatter：

```yaml
---
title: "动词或问题导向的标题"
description: "一句话说明读者读完能学到什么（不剧透全文）"
pubDate: 2025-06-15
updatedDate: 2025-06-16  # 可选，重大修订时填写
techDomain: ["记忆"]               # 技术领域，驱动首页侧栏筛选与卡片展示
product: ["Claude Design"]         # 关联产品，驱动首页侧栏筛选与卡片展示
---
```

**两个标注字段**：`techDomain` 写技术方向（讨论某方向时填）；`product` 写拆解涉及的产品（产品文时填）。不适用则写 `[]`，不要省略字段。命名保持一致：产品用官方写法，技术领域用简短中文；通常 1–2 个，避免泛化。

**标题**：8–20 字，信息密度高。**description**：回答「谁该读 + 收获什么」，不写「本文将介绍…」。

## 改稿检查清单

- [ ] 首段 30 秒内能看懂核心问题
- [ ] 每个二级标题节，删掉不影响主线则删
- [ ] 至少一处 When not to use 或等价边界说明
- [ ] 至少一个表格、图或代码块承载不可用文字替代的信息
- [ ] 结尾是原则而非摘要复读
- [ ] 技术论断有依据（实测、官方文档、可复现步骤）
- [ ] `techDomain` / `product` 与正文匹配；无关则留 `[]`

## Common Mistakes

| 错误 | 修复 |
|------|------|
| 功能清单式写作 | 改为「问题 → 方案 → 权衡」叙事 |
| 只有 What 没有 Why | 每节首句补「为何读者需要知道」 |
| 堆术语不定义 | 首次出现给一句话定义 |
| 缺少边界 | 加「不适用场景」防止误用 |
| 结尾变成目录复述 | 收束为 2–3 条可带走的决策原则 |
| 代码无上下文 | 代码前写场景，后写预期与局限 |
| `techDomain` / `product` 混填或泛化 | 技术方向归前者，具体产品归后者；只标强相关 1–2 个 |
