## Agent 为什么需要工具？

大语言模型本质上是一个「文本进、文本出」的系统。它无法直接：

- 查询实时信息（训练数据有截止日期）
- 执行代码或操作文件
- 调用外部 API
- 进行精确的数学计算

**Tool Use**（工具调用）就是为 Agent 装上「双手」，让它能够突破纯文本的局限。

## Function Calling 的工作原理

以 OpenAI 的 Function Calling 为例，流程如下：

```python
# 1. 定义工具 schema
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            },
            "required": ["city"]
        }
    }
}]

# 2. 模型决定调用哪个工具
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "北京今天天气怎么样？"}],
    tools=tools,
)

# 3. 执行工具并返回结果
tool_call = response.choices[0].message.tool_calls[0]
weather = get_weather(**json.loads(tool_call.function.arguments))

# 4. 将结果反馈给模型生成最终回答
```

## 主流方案对比

| 方案 | 特点 | 适用场景 |
|------|------|----------|
| OpenAI Function Calling | 原生支持，稳定可靠 | OpenAI 生态 |
| Anthropic Tool Use | 类似 FC，支持并行调用 | Claude 生态 |
| LangChain Tools | 框架抽象，工具生态丰富 | 快速原型 |
| MCP (Model Context Protocol) | 标准化协议，工具可复用 | 跨模型/跨平台 |

## 设计好工具的原则

1. **单一职责**：每个工具只做一件事，参数尽量简单
2. **描述清晰**：`description` 是模型选择工具的主要依据，要准确描述用途和限制
3. **错误友好**：返回结构化的错误信息，帮助模型自我纠正
4. **幂等性**：尽量让工具调用可重复执行而不产生副作用

## MCP：工具调用的未来？

Anthropic 推出的 [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) 试图标准化 Agent 与外部工具的交互方式。它的核心理念是：

> 就像 USB-C 统一了设备连接方式，MCP 统一了 AI 模型与数据源/工具的连接方式。

MCP Server 可以被任何支持 MCP 的 Client 复用，这意味着工具只需要写一次，就能在不同 Agent 框架中使用。

## 实践建议

- 从 2-3 个核心工具开始，逐步扩展
- 为工具调用添加日志，方便排查问题
- 考虑工具的权限控制和沙箱隔离
- 测试边界情况：模型传错参数、调用不存在的工具等
