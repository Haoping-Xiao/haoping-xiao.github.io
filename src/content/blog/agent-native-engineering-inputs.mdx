过去我们靠 code review 建立信任：人写代码，人读代码，重要改动由有经验的人逐行看过。

Agent 时代正在拆掉这个前提。不是因为 review 没用了，而是因为代码生成速度已经快过人类认真阅读代码的速度。继续要求人类读完所有 diff，要么让人变成瓶颈，要么让 review 变成盖章。

真正的问题变成了：**当代码生成速度远超阅读速度时，人类怎样继续保持对系统的控制和信任？**

我的答案是：质量控制的重心会从“读 diff”迁移到“管理输入”。

这里的输入不只是 prompt。它包括需求、边界、类型、测试、监控、系统历史、业务规则，以及 Agent 执行过程中暴露出来的新未知。人类的工作不再是站在合并门口说“这段代码我看过了”，而是持续塑造一个更清楚、更可验证、更能自我反馈的系统。

## 信任不住在 diff 里

传统 code review 隐含了一个假设：只要我读懂这次改动，就能判断系统是否安全。

这个假设在小改动里有用，但它从来不是完整事实。一个系统是否可信，更多来自这些东西：

| 质量来源 | 真正要问的问题 |
|----------|----------------|
| 需求 | 意图是否清楚，边界是否明确？ |
| 类型与接口 | 非法状态是否难以表达？ |
| 测试 | 真实回归发生时，它会不会变红？ |
| 监控 | 线上坏了，我们能不能及时知道？ |
| 历史信号 | bug 是否反复集中在某些模块或边界？ |
| 解释能力 | 人是否能恢复对变化的理解？ |

读 diff 只是其中一条信号，而且是一条很贵的信号。它要求 reviewer 在有限时间里重建上下文、识别业务意图、推理边界条件，还要判断测试是否真的覆盖了风险。

当 Agent 可以在短时间里生成大量可运行代码时，把所有质量责任压到“人类读完 diff”上，会越来越不现实。

更稳的做法是把人的注意力前移和后移：前移到输入质量，后移到系统信号。前面让 Agent 少猜，后面让错误更容易暴露。

## Agent 不怕写错，怕输入模糊

Agent 做错事时，我们很容易把原因归结为“模型不够强”。但很多工程错误并不是能力问题，而是输入条件问题。

需求没说清楚，Agent 只能猜。接口边界含混，Agent 会按错误理解继续实现。测试只覆盖表面，Agent 写出能过测试但行为不对的代码。业务规则只存在人脑里，Agent 根本不知道那条隐含约束。

模糊通常不是一种东西，而是几种未知混在一起：有些是你知道自己没想清楚的，有些是你心里有判断但没说出来的，还有些是你根本没意识到的历史包袱。后两类最危险，因为它们会让 Agent 在你以为“很明显”的地方开始猜。

<details>
<summary>四类未知的分类</summary>

| 类型 | 含义 | 例子 |
|------|------|------|
| Known knowns | 你知道，也已经写进 prompt | “新增一个导出按钮，格式是 CSV” |
| Known unknowns | 你知道自己还没想清楚 | “权限边界还没定” |
| Unknown knowns | 你心里有判断，但没有说出来 | “这个页面不能像营销页，要像工作台” |
| Unknown unknowns | 你没意识到的问题 | 历史包袱、隐藏依赖、线上数据异常 |

</details>

这就是 Agent-native engineering 里最重要的 mindset shift：

> Agent 的质量瓶颈，往往不是生成能力，而是人类发现未知、表达意图、收紧反馈回路的能力。

换句话说，人类的价值不是消失了，而是位置变了。过去你亲自执行，现在你要管理系统条件。过去你主要检查结果，现在你要设计让正确结果更容易出现的输入环境。

## 开始前：先发现未知

Agent 写错，很多时候是因为它遇到了你没有显式说明的未知。

开始前的目标不是把所有事情想完，而是降低 Agent 必须猜的比例。

可以让 Agent 先做 blind spot pass，让它指出你可能没意识到的风险；也可以先做 brainstorm / prototype，用几个方向逼出你的判断。如果方向已经比较明确，就让 Agent interview 你，优先问那些会改变架构、数据模型或用户体验的问题。

<details>
<summary>开始前的 prompt 模板</summary>

Blind spot pass：

```text
我准备改这个认证模块，但我不熟悉这块代码。
先帮我找出我可能没意识到的问题，并告诉我接下来应该怎么更好地 prompt 你。
```

Brainstorm / prototype：

```text
我想做一个 dashboard，但还不知道什么样算好。
先做 3 个完全不同的 HTML 原型，我看完再决定方向。
```

Interview：

```text
在实现前，逐个问我问题。
优先问那些答案会改变架构、数据模型或用户体验的问题。
```

</details>

如果你说不清想要什么，就给 references。已有代码、竞品页面、设计稿、文档、截图，甚至另一个语言里的实现，都比一句抽象描述更能减少误猜。

真正动手前，再让 Agent 写一份 implementation plan。重点不是流程漂亮，而是把数据模型、类型接口、用户可见行为、风险点这些最容易出错的决策提前摊开。

## 实现中：记录偏离

计划再好，真正进入代码也会遇到新情况。真实系统总会比 prompt 更复杂。

所以要让 Agent 在实现中记录未知、假设和偏离，而不是只交付最终代码。偏离记录的作用，是把“真实系统和计划不一致的地方”显性化，方便后续 review 和沉淀。

<details>
<summary>记录偏离的 prompt 模板</summary>

```text
如果你发现计划不适合当前代码，请选择保守方案继续，
但记录你为什么偏离原计划、遇到了什么未知、做了哪些假设。
```

</details>

这份记录很重要。它告诉你真实系统和你脑中的系统哪里不一致，哪些接口误导了 Agent，哪些规则没有被显式写出来。

你得到的不只是代码，还有一份“系统哪里还不清楚”的报告。

这会改变 review 的重点。Reviewer 不必从零开始猜 Agent 为什么这样写，而是可以沿着这些偏离点检查：这个假设是否成立？这个保守方案是否真的保守？这个未知是否应该变成文档、测试或类型约束？

## 完成后：恢复理解

Agent 做完之后，最大的问题不一定是代码有没有写对，而是：**人类是否还理解这次变化。**

如果你只是看见测试绿了、页面能跑、diff 很大，然后直接 merge，系统知识就会慢慢从人脑迁移到无人掌握的代码里。这比单个 bug 更危险。

完成后需要两个动作。

第一个是 explainer。让 Agent 说明这次做了什么、为什么这样做、解决了哪些 unknowns、排除了哪些方案、哪些风险已处理、哪些风险还存在、reviewer 应该重点看哪里。如果有 UI 或产品变化，最好附 demo、截图或 GIF。

它不是为了包装成果，而是为了让别人少走一遍你刚刚走过的认知路程。

第二个是 quiz。让 Agent 反过来考你，确认你理解关键路径、行为变化、风险和验证方式。它防止的是一种很隐蔽的失控状态：代码是 Agent 写的，测试也过了，但你其实没掌握变化。

<details>
<summary>理解检查的 prompt 模板</summary>

```text
给我一份这次改动的报告，解释关键路径、行为变化、风险和验证方式。
最后加一个 quiz，我必须全部答对才 merge。
```

</details>

你不一定要逐行读完，但你必须把判断能力拿回来。

## 完成后：读系统信号

人类当然仍然可以抽样看代码。只是重点应该从“我有没有读完每一行”转向“系统现在是否更可预测”。

几个更有用的问题是：

- 测试是不是验证真实行为，而不是只验证实现细节？
- 这次改动有没有新增 hard checks，让同类错误更难发生？
- bug 是否集中在某个模块、接口或数据边界？
- Agent 是否反复在同一个地方误解系统？
- 有没有监控能证明线上坏了会被发现？

如果某个模块反复出问题，不要只让下一个 PR review 更严格。更可能的根因是：这个模块的接口、边界或测试在误导所有使用它的人，也在误导 Agent。

这就是从“读 diff”转向“读系统”。

读 diff 看的是一次改动。读系统看的是错误为什么会在这里反复发生，以及怎样让下一次更难发生。

## 反馈后：收紧输入

每次错误都应该被用来改善输入条件。

这可能意味着把需求写得更具体，把隐含规则变成测试，把脆弱边界重构成清晰接口，用类型系统让非法状态不可表达，给关键路径加监控，或者把容易误解的 prompt、spec、文档改掉。

这就是 tightening inputs。修完这个 bug 不是结束，真正的结束是让同类错误以后更难发生。

一个团队如果只把 Agent 当成更快的打字员，会得到更多代码，也会得到更多待解释的系统行为。一个团队如果把 Agent 当成反馈放大器，就会开始看见哪些规则没写清楚、哪些边界不稳定、哪些测试没有能力失败。

差别不在 Agent 本身，而在你是否把错误转化成更清楚的输入。

## 一个可执行的 Agent 任务协议

每个 Agent 任务，可以要求五个交付物：

1. 实现结果。
2. 测试或验证证据。
3. 实现中遇到的未知、假设和偏离。
4. 给 reviewer 的 explainer。
5. 给你自己的 quiz 或理解检查。

Reviewer 也可以换一种问法：

| 少问 | 多问 |
|------|------|
| 这行代码我有没有读懂？ | 这个系统现在有没有更可预测？ |
| Agent 有没有写错？ | 我们给它的输入哪里不够清楚？ |
| 测试是不是绿的？ | 这个测试有没有能力在真实回归时变红？ |
| 这次能不能 merge？ | 同类错误以后会不会更难发生？ |

这不是放弃 review。相反，这是把 review 从“逐行阅读生成物”升级成“检查系统是否更可控”。

## 写在最后

Agent 时代的核心能力，不只是会用 AI 写代码。

更重要的是一种工作方式的转变：人类从亲自执行，转向定义问题、发现未知、收紧边界、设计反馈回路。

当 Agent 比任何人都写得快时，质量不再主要靠人类守在 gate 前逐行阅读，而靠人类持续塑造一个更清楚、更可验证、更能自我反馈的系统。

这才是 Agent-native engineering 的核心：不是让人退出工程判断，而是把人的判断力放到更高杠杆的位置。
