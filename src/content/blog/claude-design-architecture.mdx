第一次在 Claude Design 里生成简历，右侧 Canvas 往往比对话还先动起来——标题出来了，段落一行行补全，排版也在跟着变。这时候很容易误以为：Claude 写了个 HTML 文件，浏览器直接打开而已。

拆开看，并不是这样。生成的代码要在用户浏览器里执行，又不能碰 claude.ai 的 cookie 和会话；还要一边流式输出、一边更新画面。Claude Design 为此单独搭了一条链路：**Skill 定格式 → 模型按格式写 → 专用引擎解析 → iframe 里画像素**。下面按数据流走一遍。

## 整体链路

```
用户 + Skill.md
      ↓
   Claude 流式输出 Artifact 代码
      ↓
Canvas iframe（沙箱）← postMessage → 主页面
      ↓
dc-runtime / deck-stage 等引擎 → DOM → 屏幕
```

Skill 管「该长什么样」，Claude 负责产出，渲染脚本是消费者，iframe 是隔离墙。四块绑在一起，缺任何一环都没法既安全又实时。

## iframe 不是装饰，是前提

Artifact 跑在 Canvas 右侧的 iframe 里，和主站物理隔离。模型吐出来的 JS 本质是用户可控输入——如果和 claude.ai 同域，读 cookie、改会话、探测主页面 DOM 都变成可能的事。

两边只通过 `postMessage` 说话：主页面发翻页、高亮之类的指令，iframe 回传状态。不共享 DOM，也不共享存储。主站把「能闯多大祸」锁在沙箱里，自己只认消息协议。

| | 主页面 | iframe |
|---|---|---|
| claude.ai cookie | 能 | 不能 |
| 跑生成代码 | 不能 | 能 |
| 跨边界通信 | postMessage | postMessage |

如果只是预览静态 Markdown，单独 iframe 加重型渲染引擎确实偏重。但 Claude Design 要的是可交互、可流式、可执行的产出——在这个前提下，隔离不是优化项，是开工前就得定下来的边界。

## 一种产物，一套渲染脚本

HTML 简历、React 组件、PPT 幻灯片，各自走不同的 JS 管线：

| 类型 | 脚本 | 大致做法 |
|------|------|----------|
| HTML / 简历 | `support.js`（dc-runtime） | 自定义 DSL → DOM |
| React | 内置管线 | 组件树 |
| PPT | `deck-stage.js` | Web Component |

不管模板长什么样，最后都是 JS 读描述、改 DOM、浏览器画出来。差别在于中间那层「描述语言」——标准 HTML、JSX，还是下面这套 dc DSL。

## 为什么不能只写 HTML

静态 HTML 是快照，不是运行时。Claude Design 至少需要三件事，单靠 markup 做不了：

- 模型边吐 token，画面边刷新
- 同一份模板挂不同数据
- 主页面通过 postMessage 驱动翻页、重渲染

所以产物不是「`.html` 丢进 iframe」，而是「引擎能消费的格式，由引擎负责变成活的 DOM」。结构归 HTML/DSL，生命周期归 JS。

## dc-runtime 和那堆 `<x-dc>` 标签

简历和 HTML 组件走 **dc-runtime**（DC = Design Component）。它用了一套浏览器原生不认识的标签，只有 `support.js` 会解析：

| 语法 | 干什么 |
|------|--------|
| `<x-dc>` | 解析入口 |
| `<helmet>` | 往 `<head>` 塞标题、meta、样式 |
| `<sc-for>` | 循环，把静态块变模板 |
| `{{变量}}` | 插值 |

浏览器会把未知标签当普通元素略过；语义全靠运行时补。思路和 Vue 的 `v-for`、React 的 JSX 差不多——在 Web 标准上叠一层受控语法，让模型输出比裸 JS 稳定，又比纯 HTML 灵活。

PPT 另起炉灶：内部代号 **omelette**（从 `__omelette_presenting` 这类 API 名能看出来），`deck-stage.js` 走 Web Component。没有硬塞一个万能渲染器，而是哪种产物省事就哪种解释路径。

## Skill 把模型和引擎拴在一起

`Skill.md` 写死 Claude 输出时必须遵守的格式。引擎只吃符合契约的输入；模型也被训成稳定吐这种格式。

没有 Skill，生产者和消费者很容易对不齐——流式场景下尤其灾难：下一段 token 到了，解析器还不知道上一段该接到哪。Skill、DSL、引擎是，改一个得跟着改另外两个，版本得一起走。

这也解释了为什么 Claude Design 里的简历、海报往往比「随便让模型写个 HTML」整齐：不是模型突然变乖，是**输出空间被刻意收窄了**。

## 设计系统 CSS

产品还支持导入现有代码或 Figma 的 `.fig`，抽颜色、字体、间距做成可复用的设计系统 CSS。后续生成时 Claude 会参考这套约束，而不是每次从零配色。

Skill 管结构和语法，设计系统管视觉——两层契约叠在一起，少出现「能跑但很丑」或者风格每次都不一样的状况。

## 流式更新比听起来难

简历边生成边显示，大致是这样：

1. Claude 持续往 iframe 推增量片段
2. dc-runtime 用 `dcUpdate()` 接住
3. 引擎局部 patch 已挂载的组件树，而不是整页 `innerHTML` 重写

全量重绘在流式场景里又慢又闪。局部更新体验好，但引擎得吞**不完整输入**——DSL 解析器要能容忍中间态，下一段到了再修正。这比一次性渲染难不少，也是专用 runtime 存在的理由之一。

## 几个容易想偏的地方

有人把 Canvas 当成普通网页——其实是沙箱里的专用 DSL，不是随便嵌个 URL。有人以为模型直接写 React 就能跑——仍要经过对应管线和 Skill。postMessage 也不是性能优化，是跨边界唯一合法通道。

如果你的需求只是展示纯文本或代码高亮，不必复刻这整套。但要做不可信代码执行、流式 UI、多种 Artifact 形态，分层隔离 + 格式约束 + 分类型渲染，基本躲不开。

---

Claude Design 表面上是「聊天框右边多了块预览」，底下是一整套受控的生成与执行平台。搞清楚 iframe 拦什么、Skill 限什么、流式更新发生在哪一层，再去看别的产品的 Artifact 或 Canvas，会好判断很多：他们到底隔离了什么，又在哪一层做差量更新。
