当 agent 开始能写代码、读论文、帮研究员做决策时，我们关心的不是理想条件下的 recall，而是**真实工作流里能否正确推理、判断，并给出可操作的输出**。

[LifeSciBench](https://openai.com/index/introducing-life-sci-bench/)（[论文预印本](https://cdn.openai.com/pdf/b4299379-0a97-4ffa-8b9b-c3fbb299caa9/lifescibench_preprint.pdf)）用 **750 道专家题 + 19,020 条 rubric criteria** 搭了一套完整的 life science benchmark：用 workflow 定义测什么，用 rubric 定义怎么评。论文指出，现有 eval 要么偏 factual QA，要么偏 narrow 计算分析，**几乎没有 benchmark 同时具备真实工作所需的广度与深度**。

理想化 benchmark 与真实研究的 gap，论文归纳为四点：

| 考试型 benchmark 缺什么 | LifeSciBench 补什么 |
|------------------------|---------------------|
| 证据被裁剪成纯文本 QA | **复杂附件** — 图表、PDF、序列、结构文件等 |
| 指令精确、信息给全 | **情境模糊** — 模糊 prompt、不完整证据、开放式作答要求 |
| 孤立知识点、窄域技能 | **真实性** — 按科学家实际 workflow 组织，跨 domain 覆盖 |
| 单一 gold label、自动打分 | **专家审查的 rubric** — 专家写 rubric，多轮 QA 审查评分标准 |

<details>
<summary>四个 gap 分别意味着什么？</summary>

**1. 复杂附件** — 考试型 benchmark 从文献抽事实点改写成 QA（PubMedQA、BioASQ），模型只处理 prompt 文字；真实工作则要求读凝胶图、Visium h5、SMILES、PDF 等附件。LifeSciBench：53% task 需 artifact，共 1,062 个文件；有 artifact 时 GPT-Rosalind pass rate 从 44.5% 降到 28.6%。

**2. 情境模糊** — 考题边界清晰、信息给全；真实 prompt 常省略背景、目标模糊。LifeSciBench 刻意 open-ended：不只评对错，还评 caveat、假设、细节粒度是否到位。79% task 需多步推理（平均 4 步）。

**3. 真实性** — 知识点 benchmark 场景窄；真实工作横跨 evidence handling、analysis、design、translation、communication 等。LifeSciBench 用 7 workflow × 7 biological domain 组织 750 task，分类体系 用于分析强弱，而非把科研切成互斥单技能。

**4. 专家审查的 rubric** — 难点不只是「没有唯一答案」，而是**好答案的标准必须先被专家定义并审查**。LifeSciBench 每题同时写 prompt + rubric，经 ≥2 轮 expert review，重点查 question-rubric consistency；锚定 verifiable answer 或 ≥90% consensus，平均 25 条/task 的 claim-level criteria。

</details>

LifeSciBench 针对这四个 gap 给出了系统性解法，构建方法可概括为五步。

## 第一步：用 分类体系 定义「测什么」

不要从 textbook 章节出题，而是 **survey 从业者**，按他们实际做的工作聚类。LifeSciBench 得到 7 类 workflow × 7 个 biological domain，每题标注 evidence type，共 750 task——这是 **真实性** 原则的落地。

**为什么要按 workflow、domain、evidence type 三个维度划分？** 论文说，题目是围绕这套 分类体系 设计的，三个轴各管一件事：**Workflow** 管「做什么类工作」——在证据处理、分析、设计、转化等通用科研能力之间取得均衡；**Domain** 管「在哪个子领域做」——确保基因组、化学、结构、临床等专长都有覆盖，而不只测某一窄域；**Evidence type** 管「输入是什么形态」——逼模型在文本、图表、序列、PDF 等材料之间切换，同时覆盖计算与实验语境。

这里有个容易误读的点：分类体系 **不是为了**把科研工作降维成互斥的单技能。真实 task 往往混用证据解读、机制推理、决策与科学传播；三个轴的真正用途，是提供结构化视角，看清当前模型**能做和不能做**哪些生命科学工作。

<details>
<summary>7 类 workflow 与 7 个 domain</summary>

| Workflow | 测什么 |
|----------|--------|
| Evidence handling | 提取、比较、整合论文、图表、监管文件、实验输出 |
| Analysis | 定量、统计、计算或机制分析 |
| Design & optimization | 分子、assay、实验、construct、protocol 设计优化 |
| Scientific reasoning | 机制解释、假设评估、不确定性下推理 |
| Validation & operations | 排查、质控、可行性、转化风险 |
| Translation | 临床前证据 → 临床意义、试验设计、安全性 |
| Scientific communication | 面向特定受众组织、解释结论 |

Domain：Genomics · Chemistry/MedChem · Protein & Structural Biology · Molecular & Cell Biology · Assays & Screening · Bioinformatics/Comp Bio · Clinical/Translational Science

证据类型（输入侧标注）：Text/sequence · Figures/images · Tables/spreadsheets · PDFs/documents · Chemical/structure · Web links

</details>

## 第二步：Task 像「给靠谱同事提需求」

结构固定：**prompt + artifacts + rubric**。Prompt 用自然科研语言，可 intentionally open-ended；模型 **single-turn** 产出最终回答。

好 task 四条：**专家真会问 · 有 grading anchor · 能区分 caveat 遗漏 · 支持 partial credit**。

<details>
<summary>Task 结构与数据规模</summary>

```
Scientific prompt + Artifacts → Free-response → Task-specific rubric
```

| 指标 | 数据 |
|------|------|
| 多步推理 | 79%（平均 4 步） |
| 需 artifact | 53% |
| 附加文件 | 1,062 个 |
| Rubric criteria | 平均 25 条/task |

**Visium 空间转录组**（Analysis）：读 h5 附件 → 聚类注释 → 推荐治疗靶点，rubric 逐条评聚类、靶点、统计量和 caveat。**FDA 审评**（Translation）：逐项 pressure-test approval package——有核心解读，靠 reasoning 深度拉开差距。

</details>

## 第三步：专家写题 + 独立审题

173 位 PhD（≥2 年 biotech/pharma 经验）写题，453 位独立专家验题；平均 6 轮 automated review + ≥2 轮 expert review，**写题人与验题人必须分离**。

<details>
<summary>审查维度</summary>

入库前四轮审查：question-rubric consistency · scientific ambition · fact check · formatting。

</details>

## 第四步：Rubric 评过程，不只评 final answer

> 结论对但遗漏关键 assay limitation，仍是不完整；部分推理质量高但未 fully solve，也应 partial credit。

Rubric 按 **claim-level** 加权——关键 flaw 权重最高。论文 Appendix C 六原则：Specificity · Atomicity · Evaluability · Grounding · Non-redundancy · Operational usefulness。评分用 automated grader + 子集 **spot-validation**。

<details>
<summary>FDA 审评 rubric 示例与 MCQ 对比</summary>

| Criterion | Points |
|-----------|--------|
| 识别 micro-dystrophin 定量 assay 问题 | +24 |
| 解释 expression ≠ surrogate endpoint | +22 |
| 指出 biopsy / age-window confounding | +19 |
| 批评 NSAA comparator 统计 | +12 |
| 讨论 AAV durability / 免疫 / 安全 | +15 |
| 指出 patient selection 限制 | +8 |

| | MCQ | LifeSciBench rubric |
|---|-----|---------------------|
| 粒度 | 题级 0/1 | criterion 级 partial credit |
| 「差一点」 | 算错 | 50%+ score 但 pass 失败 |
| 成本 | 低 | rubric + LLM-judge + spot-check |

109 个 task：pass rate 低于 20%、rubric score 仍 ≥50%——方向对，缺关键约束。

</details>

## 第五步：Normalized Score + Pass Rate 双报告

| 指标 | 定义 | 读法 |
|------|------|------|
| **Normalized Score** | 得分 / 该题总分，task 等权均值 | partial credit、进步趋势 |
| **Pass Rate** | rubric ≥ 70% 的 task 占比 | 硬门槛：算不算做对 |

GPT-Rosalind 最强：**0.576 / 36.1%**（GPT-5.5：0.519 / 25.7%）。171 题（22.8%）零模型 pass——benchmark 远未饱和。**两个数字都要报**，且必须按论文的五维切片，否则 headline number 会误导。

<details>
<summary>论文里的 分类体系 与报告维度（别混用）</summary>

LifeSciBench 论文里其实是**两套东西**：

**① 数据集组织（Appendix B）—— task 怎么分类**

| 维度 | 含义 | 例子 |
|------|------|------|
| **Workflow** | 什么类工作 | Analysis、Translation、Design… |
| **Domain** | 哪个子领域 | Genomics、MedChem、Clinical… |
| **证据类型** | 输入侧材料形态 | PDF、figure、sequence、SMILES、web URL… |

Figure 1 概览图把分析维度简写为 workflow、domain、evidence type、**capability category**（即 rubric 按能力类型分组）。

**② 结果报告（Section 5.2）—— 性能怎么切片**

论文按 **workflow、biological domain、artifact setting、rubric category、answer format** 五个维度报告性能。

| 报告维度 | 含义 |
|----------|------|
| **Workflow** | 哪类 workflow 强/弱 |
| **Biological domain** | 哪个子领域强/弱 |
| **Artifact setting** | text-only vs 有 attached artifact |
| **Rubric category** | rubric criteria 按能力分组（如 mechanism explanation、experiment design） |
| **Answer format** | 输出侧：free text、exact numeric、sequence/structure、construct… |

**常见混淆：** 输出侧的 answer format 是**报告切片**维度，不是 Appendix B 分类体系 的第三轴；输入侧第三轴是**证据类型**（evidence type）。

</details>

<details>
<summary>维度切片数据与 benchmark 局限</summary>

- **Workflow：** Translation（0.712）和 Scientific Communication（0.718）最强；Design/Optimization 和 Analysis 最难
- **Artifact setting：** text-only 44.5% → 有 attached artifact 28.6%
- **Answer format：** sequence/structure 成功率 18–47%；generate/construct 几乎无模型间提升

**Limitation：** single-turn、self-contained task；**bench 高分 ≠ research impact**，需 deployment study 验证。

**常见陷阱：** 只测 recall · 只有 MCQ · 只看 final answer · 忽略 artifact · 单一 metric · bench 分数当 KPI

</details>

## 写在最后

1. **先 workflow，再题目** — 分类体系 决定场景覆盖，知识点分类做不出 discriminating benchmark。
2. **Rubric 是核心资产** — 评分标准本身要经过专家撰写与多轮审查，不只是为 partial credit 服务。
3. **两个数字都要报** — pass rate 说能不能用，normalized score 说离能用还有多远。

LifeSciBench 把**领域专家怎么评判一份 research memo** 翻译成了可规模化的 evaluation infrastructure。
