当 agent 开始能写代码、读论文、帮研究员做决策时，我们关心的不是理想条件下的 recall，而是**真实工作流里能否正确推理、判断，并给出可操作的输出**。

[LifeSciBench](https://openai.com/index/introducing-life-sci-bench/)（[论文预印本](https://cdn.openai.com/pdf/b4299379-0a97-4ffa-8b9b-c3fbb299caa9/lifescibench_preprint.pdf)）用 **750 道专家题 + 19,020 条 rubric criteria** 搭了一套完整的 life science benchmark：用 workflow 定义测什么，用 rubric 定义怎么评。论文指出，现有 eval 要么偏 factual QA，要么偏 narrow 计算分析，**几乎没有 benchmark 同时具备真实工作所需的广度与深度**。

理想化 benchmark 与真实研究的 gap，论文归纳为四点：

| 考试型 benchmark 缺什么 | LifeSciBench 补什么 |
|------------------------|---------------------|
| 证据被裁剪成纯文本 QA | **复杂附件** — 图表、PDF、序列、结构文件等 |
| 指令精确、信息给全 | **情境模糊** — 模糊 prompt、不完整证据、开放式作答要求 |
| 孤立知识点、窄域技能 | **真实性** — 按科学家实际 workflow 组织，跨 domain 覆盖 |
| 单一 gold label、自动打分 | **专家审查的 rubric** — 专家写 rubric，多轮 QA 审查评分标准 |

<details>
<summary>四个 gap 分别意味着什么？</summary>

**1. 复杂附件** — 考试型 benchmark 从文献抽事实点改写成 QA（PubMedQA、BioASQ），模型只处理 prompt 文字；真实工作则要求读凝胶图、Visium h5、SMILES、PDF 等附件。LifeSciBench：53% task 需 artifact，共 1,062 个文件；有 artifact 时 GPT-Rosalind pass rate 从 44.5% 降到 28.6%。

**2. 情境模糊** — 考题边界清晰、信息给全；真实 prompt 常省略背景、目标模糊。LifeSciBench 刻意 open-ended：不只评对错，还评 caveat、假设、细节粒度是否到位。79% task 需多步推理（平均 4 步）。

**3. 真实性** — 知识点 benchmark 场景窄；真实工作横跨 evidence handling、analysis、design、translation、communication 等。LifeSciBench 用 7 workflow × 7 biological domain 组织 750 task，分类体系用于分析强弱，而非把科研切成互斥单技能。

**4. 专家审查的 rubric** — 难点不只是「没有唯一答案」，而是**好答案的标准必须先被专家定义并审查**。LifeSciBench 每题同时写 prompt + rubric，经 ≥2 轮 expert review，重点查 question-rubric consistency；锚定 verifiable answer 或 ≥90% consensus，平均 25 条/task 的 claim-level criteria。

</details>

LifeSciBench 针对这四个 gap 给出了系统性解法，构建方法可概括为五步。

## 第一步：用分类体系定义「测什么」

不要从 textbook 章节出题，而是 **survey 从业者**，按他们实际做的工作聚类。LifeSciBench 得到 7 类 workflow × 7 个 biological domain，每题标注 evidence type，共 750 task——这是 **真实性** 原则的落地。

**为什么要按 workflow、domain、evidence type 三个维度划分？** 论文说，题目是围绕这套分类体系设计的，三个轴各管一件事：**Workflow** 管「做什么类工作」——在证据处理、分析、设计、转化等通用科研能力之间取得均衡；**Domain** 管「在哪个子领域做」——确保基因组、化学、结构、临床等专长都有覆盖，而不只测某一窄域；**Evidence type** 管「输入是什么形态」——逼模型在文本、图表、序列、PDF 等材料之间切换，同时覆盖计算与实验语境。

这里有个容易误读的点：分类体系**不是为了**把科研工作降维成互斥的单技能。真实 task 往往混用证据解读、机制推理、决策与科学传播；三个轴的真正用途，是提供结构化视角，看清当前模型**能做和不能做**哪些生命科学工作。

<details>
<summary>7 类 workflow 与 7 个 domain</summary>

| Workflow | 测什么 |
|----------|--------|
| Evidence handling | 提取、比较、整合论文、图表、监管文件、实验输出 |
| Analysis | 定量、统计、计算或机制分析 |
| Design & optimization | 分子、assay、实验、construct、protocol 设计优化 |
| Scientific reasoning | 机制解释、假设评估、不确定性下推理 |
| Validation & operations | 排查、质控、可行性、转化风险 |
| Translation | 临床前证据 → 临床意义、试验设计、安全性 |
| Scientific communication | 面向特定受众组织、解释结论 |

Domain：Genomics · Chemistry/MedChem · Protein & Structural Biology · Molecular & Cell Biology · Assays & Screening · Bioinformatics/Comp Bio · Clinical/Translational Science

证据类型（输入侧标注）：Text/sequence · Figures/images · Tables/spreadsheets · PDFs/documents · Chemical/structure · Web links

</details>

## 第二步：Task 像「给靠谱同事提需求」

结构固定：**prompt + artifacts + rubric**。Prompt 用自然科研语言，可 intentionally open-ended；模型 **single-turn** 产出最终回答。

好的 task 满足：**专家真会问 · 有评分锚点 · 能区分 caveat 遗漏 · 允许部分得分**。

论文附录 D 给了完整样例题。下面两例不是要报规模数字，而是用不同 workflow 展示 **task 长什么样**——prompt、artifact、rubric 如何配合，以及 rubric 为什么能评过程而不只评对错：

- **Visium 空间转录组（Analysis）**：模型读 h5 附件，做聚类注释，再推荐治疗靶点。rubric 拆开评聚类、靶点、统计量和 caveat——**有附件、多步推理、逐条给分**，对应上文「有评分锚点、允许部分得分」。
- **FDA 审评（Translation）**：逐项 pressure-test 一份批准材料，没有单一标准答案。rubric 按 assay 有效性、替代终点、对照设计等判据拉开差距——**开放作答，靠推理深度区分强弱**，也呼应第四步「不只评 final answer」。

## 第三步：专家写题 + 独立审题

173 位 PhD（≥2 年 biotech/pharma 经验）写题，453 位独立专家验题；平均 6 轮 automated review + ≥2 轮 expert review，**写题人与验题人必须分离**。

<details>
<summary>审查维度</summary>

入库前按四个维度审查（automated review 与 expert review 共用这套检查项）：

- **Question-rubric consistency**：题目和 rubric 要对得上——判据得是题目真正问了的，不能偷偷加题外要求；每条判据要能被客观评估。
- **Scientific ambition**：得像真实科研工作，测多步推理和判断，不能是背诵或简单查表。
- **Fact check**：科学结论要有附件、共识或专家判断支撑；不能过度概括，也不能表现得过分确定。
- **Formatting**：拼写、语法、格式不能干扰理解和评分。

</details>

## 第四步：Rubric 评过程，不只评 final answer

> 结论对但遗漏关键 assay limitation，仍是不完整；部分推理质量高但未 fully solve，也应给部分得分。

Rubric 按 **claim-level** 加权——关键 flaw 权重最高。写 rubric 时论文 Appendix C 要求六条原则（详见下）。评分用 **automated grader** 按 rubric 逐条给模型回答打分；再在分层抽样的子集上做 **spot-validation**——专家用同一套 rubric 独立打分，和自动评分对比相关性与 pass/fail 一致性，确认 grader 可靠。

<details>
<summary>Rubric 撰写六原则</summary>

- **Specificity**：每条判据描述回答的一个具体属性，不能含糊笼统。
- **Atomicity**：一条判据只评一个主张、计算、决策或约束。
- **Evaluability**：单凭模型回答就能判断满足与否，不需要额外脑补。
- **Grounding**：判据要能从 prompt、附件、公认事实或专家共识中找到依据。
- **Non-redundancy**：同一件事不重复计分，除非评的是不同侧面。
- **Operational usefulness**：不只评孤立正确性，还要评对这道科研决策有没有实际用处。

</details>

## 第五步：Normalized Score + Pass Rate 双报告

| 指标 | 定义 | 读法 |
|------|------|------|
| **Normalized Score** | 得分 / 该题总分，task 等权均值 | 部分得分、进步趋势 |
| **Pass Rate** | rubric ≥ 70% 的 task 占比 | 硬门槛：算不算做对 |

GPT-Rosalind 最强：**0.576 / 36.1%**（GPT-5.5：0.519 / 25.7%）。171 题（22.8%）零模型 pass——benchmark 远未饱和。**两个数字都要报**，且必须按 workflow、domain 等维度切片，否则 headline number 会误导。

## 写在最后

1. **先 workflow，再题目** — 分类体系决定场景覆盖，知识点分类做不出 discriminating benchmark。
2. **Rubric 是核心资产** — 评分标准本身要经过专家撰写与多轮审查，不只是为部分得分服务。
3. **两个数字都要报** — pass rate 说能不能用，normalized score 说离能用还有多远。

LifeSciBench 的核心产出不是 750 个标准答案，而是**专家怎么判一道生命科学任务做得好不好**——写进 rubric、能逐条打分、能按 workflow 切片分析。
