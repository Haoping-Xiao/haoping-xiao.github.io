当 coding agent 从「偶尔调 API」变成「常驻终端、改仓库、跑 shell」时，工程问题不再是「能不能打出一条 log」，而是：**同一轮 turn 里，产品要看行为漏斗、SRE 要看 P99 延迟、安全要看敏感链路、用户报 bug 时要能挂上现场——这些信号能不能各走各的路，又不让业务代码变成 if/else 选厂商？**

这篇用 Codex 与 Claude Code 两条已落地的管线做对照，归纳当下 coding agent 可观测性的常见分法：**收什么、埋在哪、怎么导出**。读完后你能判断：该用 OTEL 还是自研 event API、默认开什么、以及哪里不该混成一条管道。

## 先划边界：五类信号，五个问题

Coding agent 的可观测性，本质上是在回答五个不同的问题：

| 信号 | 回答的问题 | 典型内容 | 谁最关心 |
|------|-----------|---------|---------|
| **Metrics** | 快不快、稳不稳、贵不贵？ | 延迟、次数、token 用量、错误率 | SRE、成本、容量规划 |
| **Events** | 用户干了什么、漏斗走到哪？ | turn 完成、tool 调用、插件启用 | 产品、增长、功能采用率 |
| **Trace** | 这一轮 agent loop 怎么跑的？ | turn → 推理 → tool → 再推理 的 span 树、每步耗时 | 性能排障、loop 卡在哪 |
| **Logs** | 会话里发生了什么结构化事实？ | turn 状态变迁、配置快照 | 值班、审计、事后复盘 |
| **Feedback** | 用户主动报的个案 | bug 描述、截图、附件 | 客服、质量、个案追踪 |

**记一句：** Metrics 看曲线，Events 看行为，**Trace 看 agent loop 怎么转**，Logs 看事实，Feedback 看个案。

## 行业共识：埋点一次，导出分流

成熟做法的共同点是：**业务只表达「发生了什么」，去向由 Exporter / Sink 配置决定**，而不是在 `run_turn` 里写 `if statsig ... else sentry ...`。

```
业务代码（run_turn / run_tool / hooks）
    │
    ├─ SessionTelemetry / OTEL Meter ──OTLP──► Metrics 后端（曲线、告警）
    │
    ├─ agent loop span（turn/tool/llm）─OTLP──► Trace 后端（默认常关）
    ├─ structured log ───────────────OTLP──► Logs 后端（默认常关）
    │
    ├─ AnalyticsFact / logEvent() ──HTTP──► 自研 Event API（行为分析）
    │
    ├─ CodexFeedback / errorLogSink ───────► Sentry / 本地 jsonl（个案与错误）
    │
    └─ 本地 store ─────────────────────────► 本机文件 / SQLite（调试回放）
```

<details>
<summary>OTEL / OTLP 前置：三个词、三个 URL</summary>

| 词 | 人话 |
|----|------|
| **OTEL** | 开放标准 + SDK，在进程内整理 span / metric / log |
| **OTLP** | 发送协议，通常 HTTP POST 到后端 |
| **Resource** | 每批数据的「身份证」：`service.name`、`environment`（三种信号共用） |

三条出口，三个 URL：

```
/v1/traces   ← agent loop 的 span 树（turn 包 model 包 tool）
/v1/metrics  ← counter / histogram
/v1/logs     ← 结构化 log 事件
```

在 coding agent 里，Trace 的 span 通常长这样：

```
session span
└─ turn span
   ├─ llm span（ttft、token）
   ├─ tool:shell span
   ├─ llm span
   └─ tool:grep span
```

进程启动时配一次 Provider，在 `run_turn` / `run_tool` 入口挂 span；换后端改 Exporter，不动业务代码。Codex（Rust）走 `tracing::info_span!` → `tracing-opentelemetry` → OTEL SDK → OTLP；Claude Code 的 `sessionTracing` 同理，是同一套心智模型。

</details>

## 案例：Codex 与 Claude Code

两家做法不同，但都是「埋点一次、导出分流」。下面只保留架构。

### Codex

| 信号 | 默认去向 |
|------|---------|
| Metrics | Statsig（OTLP） |
| Events | 自研 HTTP API |
| Trace / Logs | 不上报（可自配 OTLP） |
| Feedback | Sentry |
| 本地 | 文件 / SQLite / rollout |

```
run_turn / run_tool
    ├─ SessionTelemetry ──► OTEL Metrics ──OTLP──► Statsig
    ├─ agent loop span ───► OTEL Trace  ──OTLP──► 可配（默认关）
    ├─ tracing event ─────► OTEL Logs   ──OTLP──► 可配（默认关）
    ├─ AnalyticsFact ─────► POST /codex/analytics-events/events
    └─ CodexFeedback ─────► Sentry
```

远端默认只开 Metrics；Events、Feedback、本地各走一路。

### Claude Code

两条管道，名字分开：

**Analytics（默认开）** — 行为事件，`logEvent()` 广撒网：

```
hooks / tools / api
    │ logEvent('tengu_xxx')
    ▼
analytics/index → sink（采样）→ metadata（enrichment）
    ├─► Datadog Logs（白名单）
    └─► POST /api/event_logging/batch → BigQuery
```

**Telemetry（需 `CLAUDE_CODE_ENABLE_TELEMETRY=1`）** — OTEL metrics / trace / logs：

```
run_turn / run_tool
    ├─ OTEL Meter  ──┐
    ├─ OTEL Span   ──┼─► instrumentation.ts ──► OTLP / BigQuery / beta 端点
    └─ OTEL Logger ──┘
```

Trace 走 `sessionTracing`，记 session → turn → llm / tool 的 loop 树。错误另落本地 `~/.claude/errors/*.jsonl`，不上报。

**对比：** Codex 默认只开 Metrics 远端；Claude Code 默认开 Analytics，Telemetry 需手动开。两家都把 loop Trace 留给 opt-in。

## 自己设计时

- **默认开关**：Metrics + Events 先开；loop Trace、全量 Logs 默认关，用户或企业自配。
- **埋点位置**：`run_turn` / `run_tool` 旁注入；版本、环境等 enrichment 集中一层，业务只传事实。
- **导出分流**：OTEL 收 metrics / trace / logs，Events 走 HTTP batch，Feedback 走 Sentry；换后端改 Exporter，不改业务代码。

## 常见陷阱

| 陷阱 | 后果 | 修复 |
|------|------|------|
| 用 Metrics 后端存行为事件 | 无法漏斗分析、label 爆炸 | Events 走自研 API 或 Logs 型分析库 |
| 业务里按环境 if 选 Datadog/Statsig | 测试困难、漏报 | Exporter 配置分流 |
| loop Trace 默认全量上报 | prompt/路径泄露、账单爆炸 | 默认关 + 采样 + 高敏 loop 单独端点 |
| 把 Events 当 Trace 用 | 只有离散事实、没有 loop 结构 | loop 步骤挂 span，业务里程碑走 Events |
| 没有本地留档 | 用户报 bug 无现场 | jsonl / SQLite / rollout 回放 |
| `logEvent` 与 OTEL log 混为一个概念 | 团队沟通成本、重复埋点 | 文档里命名两条管道（Analytics vs Telemetry） |

## 写在最后

1. **五类信号五道题**——先弄清谁在问什么，再选 Metrics、Events、Trace、Logs 或 Feedback；不要一个 Dashboard 答所有题。
2. **埋点一次、导出分流**——业务表达事实，OTEL + HTTP sink 决定去向；换厂商改配置，不改 `run_turn`。
3. **默认保守、可加深**——远端默认 Metrics（+ 行为 Events）常见；**agent loop 的 Trace** 和全量 Logs 交给用户或企业自配；本地文件是安全网。

Codex 与 Claude Code 路径不同，但都指向同一件事：**agent 的可观测性不是「接满监控大盘」，而是把产品行为、SLO 数值、个案反馈和本地复现拆成可独立演进的几条管**——你在设计自己的 coding agent 时，按这张表对齐默认开关和 schema，比先选.vendor 名单更重要。
